# Scatter Core — Synthesis

**Status:** Synthesis of `CORE_HTTP_CONTRACT.md` (thesis, 2026-04-20 morning) and the 10-point antithesis pushback (2026-04-20 mid-morning). This is the working spec going forward. The two prior documents are preserved as the record of how this position was reached.

## What this machine is

Scatter is a **single-door, capability-based, journal-as-brain, cluster-real-from-day-one, two-model-verified, license-neutral, location-independent local AI OS** whose ship test is whether a child can learn something and their parent can see it.

That sentence is load-bearing. Each clause is a structural commitment that came out of the dialectical pass. Below, each one as a discipline.

---

## The ten disciplines

### 1. One door, measured

Every component — including the GUI host, the cluster, the voice daemon, Studio, Academy — talks to Core through `/v1/` HTTP on `127.0.0.1:3333`. No in-process exception. No internal-vs-external distinction. The contract is the boundary, period.

The latency tax is *measured* per release. A `latency.jsonl` file in `~/.scatter/` records p50/p99 for every endpoint. Any path that exceeds 10ms p99 may earn a documented fast lane — but the fast lane is named in this document and tested separately, never silent.

**Why this changed:** The thesis exempted 27 in-process callsites because I assumed loopback HTTP was expensive without measuring. A self-bypassing contract is a contract in name only.

### 2. Capabilities, not users

Identity is not a login. It is a URL.

A `Capability` is a long opaque string (32 bytes, URL-safe) bound to: a kind (`actor` / `observer`), a scope (a session id, a user id, or `*`), an expiry, and an optional revocation reason.

```
GET  /v1/capabilities/me                    → describe the cap on this request
POST /v1/capabilities/issue                 → operator-only; mints a new cap
POST /v1/capabilities/revoke                → revoke by id or by scope
```

Every request carries `Authorization: Cap <capability>`. Scatter trusts no presence — only capabilities. Loopback no longer implies privilege.

**A parent observing a child** is given an `observer` capability for the child's user scope. The parent opens the URL on their laptop / phone / wherever. They see what was journaled under the child's user. They cannot act as the child. Revoke = paste the cap into `/v1/capabilities/revoke`.

**Why this changed:** The thesis assumed single-machine, single-operator. Children + parents do not live on the same machine. Capabilities are a 50-year-old idea that handles this exactly without inventing an account system.

### 3. Journal as primary brain

`~/.scatter/journal.jsonl` is not just an audit trail. It is *the corpus that is most relevant to this user.*

`/v1/corpus/search` returns hits in this order:
1. **Personal**: this user's journal entries matching the query.
2. **Local**: shared community / classroom journal entries (when present).
3. **External**: the installed external corpus (CK-12, Eureka, etc.).

Every `/v1/teach/*` response writes back to the journal — what was asked, what was retrieved, what was answered, what was checkpointed. The child's brain grows because the brain *is* the journal of their lived learning.

**Why this changed:** Read-only corpus = read-only brain = "child uses someone else's brain." The mission requires the opposite.

### 4. Cluster real from day one or absent

Phase 0 includes Ollama on each Pi worker with one model loaded (`llama3.2:3b` is the v0 default for size). `destination: "cluster"` in `RoutingDecision` works on day one or `RoutingDecision.destination` does not include `"cluster"`.

No facade. No "routing metadata that nobody serves." The cluster is the cluster — it routes *and* it inferences — or it is not in v0.

The cluster manifest:
```
~/.scatter/cluster.json = {
  "head_node": "localhost",            # or "pi-head.tail-net.ts.net" for school install
  "workers": [
    {"host": "pi-1", "model": "llama3.2:3b", "capabilities": ["inference"]},
    {"host": "pi-2", "model": "llama3.2:3b", "capabilities": ["inference"]},
    {"host": "pi-3", "model": "llama3.2:3b", "capabilities": ["inference"]},
    {"host": "pi-4", "model": "llama3.2:3b", "capabilities": ["inference"]}
  ]
}
```

**Why this changed:** A v0 contract that defers the cluster is a v0 contract that defers the product story. The shipping bar is "AI on your own hardware actually working," not "the wiring exists for it to work eventually."

### 5. Two-model verification on every teach response

Every `/v1/teach/*` response that returns a fact a child will rely on is generated by Model A and verified against the same retrieved citations by Model B. A and B are different models (e.g., Qwen + Llama, or local + Claude) — the choice is in `~/.scatter/teach.json`.

```
LessonResponse = {
  primary: { content, citations, model },
  verification: {
    agree: bool,
    notes: str,                 # what the verifier flagged or confirmed
    model: str
  }
}
```

If `verification.agree == false`, the operator (parent / teacher / older child) is shown both responses. The child sees a "your grown-up is checking this one" message. No silent confidence in a wrong answer.

**Why this changed:** A leak-free test catches network leaks. Nothing in the thesis caught *truth* leaks. For a child, truth leaks are the worst kind.

### 6. License-neutral OS, operator-chosen corpus

Scatter ships with **no corpus bundled**. The operator runs `scatter corpus install <id>` to add one. The corpus's license terms then apply to that installation.

Recommended corpora and what they enable:
- **`ck12-k5-math`** (CC BY-NC) — non-commercial only. Engineering proving ground.
- **`eureka-k5`** (CC BY-NC-SA) — non-commercial only. Pedagogical primary.
- **`ccss-public`** (public domain) — commercial-OK. The standards themselves; sparse but unrestricted.
- **`openstax-k12`** (CC BY) — commercial-OK with attribution. Stronger for 6-12.

Scatter the OS is MIT and stays MIT. The commercial posture of any *deployment* is determined by the corpora the operator installed. Scatter Schools as a paid offering is structurally possible — the school just has to install commercial-OK corpora.

**Why this changed:** Bundling NC corpus would foreclose paid Scatter forever. That decision has to be the operator's, not the OS author's.

### 7. Head node is declared, not assumed

`~/.scatter/cluster.json` (above) has `head_node`. Default: `localhost`. School install: a Tailscale address of the designated Pi at the front of the room. Same contract, same `/v1/` shape, different host.

The GUI host (`scatter/server.py`) becomes a *client* of Core like everything else. `scatter_core.py` exposes the contract; `server.py` is the GUI's HTTP client. When the head moves to a Pi, the laptop's GUI just points at the Pi.

**Why this changed:** "Laptop is the universe" doesn't survive the school deployment story. The contract has to be portable across head nodes.

### 8. Two ship tests

**Engineering ship test (necessary):** All five Phase 0 endpoints round-trip end-to-end with measured latency under budget. Pre-commit hook passes. Two-model verification fires on a known-wrong test prompt and disagrees.

**Product ship test (sufficient):** A child user can ask Scatter to teach them long division. They receive a citation-grounded lesson from the operator-installed corpus. They work through three checkpoints. A parent, holding only an observer capability URL, sees the session afterward from a different device. No part of this requires Ryann to be present.

Phase 0 is not done until both pass. The engineering test is run in CI; the product test is run by Ryann + one trusted parent + one child.

**Why this changed:** "One Studio mode makes a Claude call" proves wiring. It does not prove the product exists.

### 9. Watts honestly labeled

Until a calibrated USB power meter sits on the laptop and per-Pi draw is measured, every watt number in the UI is rendered as `~12J (estimated)`. The estimation method is documented at `/v1/about/watts-method`. Calibration is a Phase 0 task: buy a $30 meter, measure one model's true draw under load, divide by tokens, publish the resulting per-token coefficient.

After calibration, the label drops the `~`. Until then, the `~` is the legibility.

**Why this changed:** Confident decimals disguising assumptions are the opposite of legibility, even with no data leaving the machine.

### 10. Architectural tests are real greps, in pre-commit

Five greps, named, shipped in `scatter/hooks/pre-commit`:

1. **No external network imports outside `scatter/api.py`.** (existing)
2. **No `import scatter_core` outside `scatter-system/scatter/`.** (new — enforces single door)
3. **No `~/.scatter/` filesystem path strings outside `scatter_core.py`.** (new — enforces storage isolation)
4. **No `/v1/messages` request body lacking a `decision_id`.** (new — enforces routing accountability; checks call sites in client repos)
5. **No `/v1/teach/*` response without a `citations:` field.** (new — enforces grounded teaching)

If a grep fires, commit fails. No `--no-verify`. The architecture is enforced by code, not by intent.

**Why this changed:** "Architectural test" is rhetorical until the grep exists.

## What ships in Phase 0

A **two-week** Phase 0, ending with both ship tests passing. The product test gates the release.

**Week 1 — substrate:**
- D1: Capability scheme. `/v1/capabilities/*` endpoints. `Authorization: Cap` parsing. Operator capability for `default` user. Existing single-user behavior unchanged from outside.
- D2: Sessions move from in-memory dict to `~/.scatter/sessions/<user>/<id>.json` with ETag. Concurrent-write test in CI.
- D3: Cluster manifest. `~/.scatter/cluster.json`. `/v1/about/cluster` reports topology. Ollama installed on pi-1 with `llama3.2:3b`.
- D4: Ollama on pi-2, pi-3, pi-4. `scatter-cluster` orchestrator routes to a worker, worker runs Ollama, returns content.
- D5: `/v1/route` returns `RoutingDecision`. `/v1/messages` enforces `decision_id`. The five greps land in pre-commit. Watt meter ordered.

**Week 2 — teaching loop + product test:**
- D6: Corpus ingestion CLI: `scatter corpus install ck12-k5-math`. CK-12 K-5 math ingested into `~/.scatter/corpus/ck12-k5-math/`. Concept-level + paragraph chunks. Local embeddings via `nomic-embed-text` running in Ollama.
- D7: `/v1/corpus/search` returns journal-first, then external. `/v1/teach/request` returns a `LessonPlan` grounded against retrieved citations.
- D8: Two-model verification. `/v1/teach/checkpoint` runs Model A + Model B on the same citations. Disagreement surfaces to operator capability.
- D9: Parent observer capability flow. Issue cap, open URL on phone, see child's session journal.
- D10: Run the product test with one trusted parent + one child. Long division, three checkpoints, parent verification. Iterate.

**Phase 0 ships when both tests pass.**

## What this is not

This synthesis is not a feature list. It is a set of structural commitments that, together, are a **different machine** from the thesis. The thesis was "ship the contract." The synthesis is "ship the contract whose implementation cannot drift away from the mission."

It is not the final spec. v1 will reveal things v0 cannot yet see. The Scatter Method requires preserving the next round of dissent, not pretending this synthesis is final.

It is not safe. Two-model verification reduces but does not eliminate wrong answers. Capability URLs can be leaked. The cluster can fail. The corpus can be wrong about something. None of this excuses shipping. All of it is logged.

It is not infinite scope. Phase 0 is two weeks. Everything beyond is named in the prior `CORE_HTTP_CONTRACT.md` "deferred" section and stays deferred until Phase 0 ships.
